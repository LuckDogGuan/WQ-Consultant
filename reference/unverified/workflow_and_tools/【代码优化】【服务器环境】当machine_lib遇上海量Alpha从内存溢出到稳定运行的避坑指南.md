原本的machine_lib.py的逻辑主要是先将所有的alpha生成，加入一个list。然后，会将这个list读入内存，然后再一批一批调用api去实现回测。早期的时候，在本地PC因为性能较高，内存较大，没有发现什么问题。可当我将回测环境部署到云服务器的时候，问题就显现了。大部分云服务器都是2C2G（2核芯2G内存），愿意花点钱就是4C4G的。相比较本地的PC性能差异就很大。此时如果你选择的数据集字段较多，或者探索alpha的空间较大，服务器就会直接卡死。原因就是，原来的machine_lib.py将所有的alpha list都一次性的读入内存。2C2G（2核芯2G内存）的服务器挂个SSH的vscode后台都会卡。哈哈哈哈。不知道是不是jd服务器的问题。2C2G也不太容易上数据库消息队列啥的。后来我斥巨资（几十块）上了腾讯的4C4G。好太多了。vscode远程很流畅。（非广告，只是分享一下经验，仅供大家参考）于是，我对machine_lib.py进行了修改。核心就是用yield生成器，一次用多少生成多少。这样就算最低配置的服务器2C2G，也不会出现，选字段多的数据集，就可能直接卡死的情况出现。这种技术路线从根本上解决了这个问题。因为其实只有一阶主要可能涉及大量的alpha
